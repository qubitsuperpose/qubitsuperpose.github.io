# 2. Compute resources and resource management
## 1. 리소스 모델 및 고전적 리소스

이번 섹션에서는 노트북에 적용할 수 있고 슈퍼컴퓨터로 확장할 수 있는 컴퓨팅 환경에 대한 틀을 제시합니다. 이번 섹션을 마치면 컴퓨팅 환경의 기본 구성 요소와 이들이 서로 어떻게 연관되어 있는지 이해하게 될 것입니다. 이 모든 내용은 Iskandar Sitdikov가 다음 영상에서 설명합니다.

### 1.1 리소스 모델

모든 고전적 컴퓨팅 환경은 애플리케이션을 효율적으로 실행하기 위해 함께 작동하는 여러 상호 연관된 리소스로 구축됩니다. 주요 리소스는 일반적으로 다음을 포함합니다:

* CPU(중앙 처리 장치): CPU는 프로그램 명령을 해석하고 실행하는 핵심 처리 장치입니다. 논리, 산술 및 제어 작업을 처리하며, 본질적으로 시스템의 "두뇌" 역할을 합니다.

* CPU 캐시(L1, L2, L3): 시스템에서 가장 빠르며 CPU 코어 내부 또는 매우 가까운 곳에 직접 구축됩니다. CPU가 즉시 필요로 하는 매우 작은 부분의 데이터와 명령을 저장합니다. 서로 다른 레벨(L1, L2, L3)은 절충안을 나타냅니다. L1은 가장 작고 가장 빠른 반면, L3는 가장 크고 가장 느리지만, 여전히 RAM보다 자릿수 단위로 더 빠릅니다.

* RAM(랜덤 액세스 메모리): 프로그램 명령과 활성 사용 데이터에 대한 크고 임시적인 저장 공간을 제공하는 휘발성 메모리입니다. 이를 통해 CPU는 느린 저장 장치에 지속적으로 의존하지 않고도 실행 중에 필요한 정보에 빠르게 액세스할 수 있습니다.

* 스토리지(로컬 및 네트워크 기반): 스토리지는 시스템 전원이 꺼진 상태에서도 데이터와 소프트웨어를 유지하여 대규모 데이터 세트와 애플리케이션에 대한 장기적인 영구성을 제공합니다. 고성능 컴퓨팅에서 스토리지 솔루션은 속도와 신뢰성을 모두 갖추어 방대한 양의 과학적 또는 분석적 데이터를 처리해야 합니다. 로컬 스토리지에는 솔리드 스테이트 드라이브(SSD)와 하드 디스크 드라이브(HDD)가 포함되며, SSD는 낮은 지연 시간과 높은 처리량 덕분에 선호됩니다. 대규모 데이터 처리를 위해 병렬 파일 시스템, 공유 네트워크 스토리지 및 객체 기반 시스템은 많은 컴퓨팅 노드에서 빠른 액세스를 가능하게 하는 한편, 클라우드 및 아카이브 스토리지 계층은 장기 보존과 확장성을 지원합니다.

* GPU(그래픽 처리 장치): 초기에는 그래픽 렌더링을 위해 설계되었지만, 현대의 GPU는 강력한 병렬 프로세서입니다. 딥러닝, 물리 시뮬레이션, 빅데이터 분석과 같이 많은 동시 계산을 필요로 하는 작업을 처리하는 데 널리 사용됩니다. GPU가 CPU를 대체하는 것은 아니라는 점에 유의하는 것이 중요합니다. CPU는 상위 수준의 프로그램 논리를 지시하고 GPU는 고도로 병렬화된 단계를 가속화합니다.

* 연결/버스: CPU, 메모리, 스토리지 및 주변 장치를 연결하는 통신 경로입니다. 버스는 시스템 부품 간의 데이터 전송과 조정을 가능하게 하여 컴퓨팅 환경 전반에 걸친 원활한 통신을 보장합니다. HPC 시스템에서 CPU, GPU 및 스토리지 장치와 같은 구성 요소는 빠른 데이터 교환을 가능하게 하는 고속 인터커넥트로 연결됩니다. GPU는 일반적으로 효율적인 통신을 위해 여러 데이터 레인을 갖춘 표준 인터페이스인 PCIe를 통해 시스템에 연결됩니다. 더 높은 성능을 위해 NVLink는 GPU 간 또는 GPU와 CPU 사이에 직접적인 고대역폭 링크를 제공하여 지연 시간을 줄이고 병렬 워크로드를 가속화합니다.

* 파일 시스템: 파일 시스템은 스토리지 장치의 데이터를 구성합니다. 파일을 저장, 검색 및 관리하기 위한 구조를 제공하여 프로그램과 사용자가 일관되고 논리적인 방식으로 정보에 액세스할 수 있도록 합니다.

각 유형의 리소스에는 자체적인 성능 관련 측정 단위가 있습니다. 예를 들어, CPU는 일반적으로 "코어"와 "클럭 속도"로 측정됩니다. 노트북을 구매할 때 사양에는 보통 코어 수가 포함됩니다. 이와 유사한 개념이 데이터 센터의 컴퓨팅 노드에도 적용되어, 각 노드는 특정 수의 코어와 연관됩니다. 여러 유형의 리소스(CPU, GPU, 심지어 QPU까지)를 포함하는 컴퓨팅 환경은 **이종 컴퓨팅 환경**이라고 합니다. 이러한 설정은 각 프로세서 유형의 강점을 활용하여 다양한 워크로드를 더 효율적으로 처리합니다. 예를 들어, CPU는 일반적인 작업에 사용되고 GPU는 병렬 처리에 사용됩니다. 리소스 관리 및 스케줄링의 맥락에서, 특히 이종 컴퓨팅 환경의 경우 여기에 설명된 단위에 추가적인 측정 단위가 필요할 수 있습니다.

메모리의 경우 측정 단위는 메가/기가/테라바이트가 됩니다.

그래픽 카드 및 기타 가속기의 경우 측정 단위는 상황에 따라 다릅니다. 실제 컴퓨팅 성능은 처리 코어 수, 메모리 크기 및 메모리 대역폭과 같은 세분화된 지표로 측정되지만, 클러스터 리소스 또는 작업 스케줄링에 대한 높은 수준의 논의에서 GPU 및 유사한 가속기는 할당된 전체 장치 수(예: 3개의 GPU)로 장치 수준에서 정량화될 수 있습니다.

네트워크/연결성/버스는 컴퓨팅 구성 요소 간에 데이터가 얼마나 빠르게 전송되는지를 결정하므로 모든 컴퓨팅 인프라의 중요한 측면입니다. LPU에서 CPU 캐시, RAM, PCI 카드, 네트워크로 연결된 장치에 이르기까지 이 모든 것이 통신이며, HPC를 위한 고도로 최적화된 알고리즘을 설계하려면 이에 대한 정확한 개념적 모델을 갖추는 것이 중요합니다.

### 1.2 고전적 리소스 확장

고성능 컴퓨팅(HPC)은 이러한 고전적 리소스를 확장하여 더 빠른 처리 시간을 달성하거나 동시에 처리할 수 있는 데이터의 양을 늘리는 것입니다(예: 검색할 수 있는 해 공간의 크기를 늘리기 위해). 이는 다음 방법을 통해 달성할 수 있습니다:

* 수직 확장: 단일 리소스의 성능을 높이는 것으로, 예를 들어 더 강력한 CPU를 사용하거나 여러 컴퓨팅 리소스를 캡슐화한 컴퓨팅 클러스터의 단위인 단일 물리적 노드 내에서 더 많은 메모리를 추가하는 것입니다.

* 수평 확장: 단일 노드 또는 더 일반적으로는 여러 노드에서 함께 작동하도록 여러 CPU나 GPU와 같은 더 많은 리소스를 추가하여 분산 컴퓨팅을 가능하게 하는 것입니다.

![단일 노드 내에 메모리와 같은 더 많은 리소스를 배치하여 리소스를 수직으로 확장하고, 서로 다른 리소스 유형을 포함하는 연결된 노드 수를 증가시켜 수평으로 확장하는 것을 보여주는 이미지.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/compute-resources/resource-scaling.svg)

이번 섹션의 확장 개념 중 일부는 양자 컴퓨팅 리소스에 대한 다음 섹션에도 적용될 것입니다. 양자 리소스의 다른 일부 측면은 새로운 방식으로 정량화될 것입니다.

> 정답은 여러 가지가 있을 수 있습니다. 수직 확장은 특히 고정된 양의 리소스가 필요한 예측 가능한 워크로드가 있는 경우 더 단순한 경우가 많습니다. 하지만 컴퓨팅의 기본 단위를 수평 확장만큼 쉽게 분해할 수 없기 때문에 업그레이드 비용이 더 많이 들 수 있습니다. 수평 확장은 관리가 더 복잡하며 때때로 노드 간 연결과 관련된 어려움이나 지연 시간이 발생할 수 있습니다. 하지만 변화하는 리소스 요구 사항에 훨씬 더 적응력이 뛰어나며 업그레이드가 필요할 때 모듈식입니다.


## 2. 새로운 유형의 리소스: QPU (Quantum Processing Unit)

이 섹션에서는 새로운 유형의 리소스인 양자 리소스를 소개하고, 그 정의, 측정 단위, 그리고 클래식 인프라와의 연결성을 살펴보겠습니다.

### 2.1 QPU의 정의

* 양자 처리 장치 (QPU): QPU는 실 가능한 양자 명령어 세트 또는 양자 회로를 받아들이고 정확한 답변을 반환하는 역할을 하는 모든 하드웨어를 포함합니다.

이는 하나 이상의 양자 칩 (예: Heron), 희석 냉동기 내의 양자 증폭기와 같은 여러 추가 구성 요소, 제어 전자 장치, 그리고 메모리에 명령어와 파형을 보관하고, 결과를 축적하며, 향후 오류 수정 디코딩과 같은 작업을 위해 필요한 클래식 컴퓨팅을 포함합니다. 이러한 작업을 수행하려면 희석 냉동기가 필요하지만, 동일한 냉동기 내에 여러 QPU가 있는 경우를 허용하기 위해 이 정의에서는 희석 냉동기를 제외합니다.

* 양자 컴퓨터: 양자 컴퓨터는 QPU와 런타임 환경을 호스팅하는 클래식 컴퓨팅으로 구성됩니다.

* 런타임 환경: 프로그램을 실행할 수 있게 해주는 하드웨어와 소프트웨어의 조합입니다.

### 2.2 양자 회로의 레이어

클래식 컴퓨팅과 양자 컴퓨팅 모두에서 프로세스는 순차적으로 또는 병렬로 실행될 수 있습니다. 큐비트는 클래식 비트에 비해 풍부한 상태 공간을 가지기 때문에, 때로는 여러 단일 큐비트 게이트가 큐비트에서 순차적으로 실행되는 것이 의미가 있습니다 (예: `R_x` 게이트 다음에 `R_z` 게이트가 오는 경우). 큐비트 간의 얽힘은 양자 컴퓨팅에 중요하기 때문에, 양자 회로가 많은 큐비트에 걸쳐 작용하는 얽힘 게이트 세트를 갖는 것도 흔합니다. 이러한 요인들과 다른 요소들로 인해 양자 회로에서 개별 게이트 연산 규모로 병렬로 행될 수 있는 프로세스를 식별하는 것이 일반적입니다. 클래식 컴퓨팅에서는 비트 수준 병렬 처리도 가능하지만 게이트 수준에서는 덜 일반적으로 고려되며, 더 큰 규모에서 병렬 및 순차 프로세스를 언급하는 것이 더 일반적입니다.

양자 컴퓨팅에서는 동시에 모두 실행될 수 있는 게이트의 "레이어"를 언급합니다. 많은 응용 프로그램에서 모든 큐비트에 대해 회전 세트를 수행한 다음 큐비트 쌍 사이에 얽힘 게이트를 수행하는 것이 유용합니다. 이러한 맥락에서 "회전 레이어" (`R_x`, `R_y` 및/또는 `R_z`와 같은 게이트의 레이어) 와 "힘 레이어" (`CNOT` 게이트가 있는 레이어와 같음) 를 언급합니다. 회로에서 레이어의 수는 "회로 깊이"이며, 더 큰 깊이는 노이즈와 오류의 중첩된 레이어가 더 많다는 것을 의미하므로 중요한 척도입니다.

레이어가 배리어를 사용하여 정렬되지 않은 경우 게이트 레이어를 시각적으로 식별하기 어려울 수 있습니다. Qiskit에서 배리어는 양자 회로에서 시각적 구분자 역할을 하고 컴파일 중에 제약 조건으로 작용하는 명령어입니다. 회로를 그릴 때와 실행할 때 모두 게이트는 배리어를 넘어 이동되지 않습니다. 이는 동적 결맞음 제거 (dynamical decoupling) 와 같은 맥락에서 중요할 수 있는데, 여기서는 특정 오류 유형을 억제하기 위해 항등식으로 단순화되는 게이트를 의도적으로 구현합니다. 동적 결맞음 제거에 대한 자세한 내용은 [이 가이드](https://quantum.cloud.ibm.com/docs/guides/error-mitigation-and-suppression-techniques)를 참조하십시오. 배리어의 시각적 효과를 위해, 첫 번째는 레이어 정렬을 강제하는 배리어가 없고 두 번째는 레이어 정렬을 강제하는 배리어가 있는 동일한 회로의 이 두 이미지를 비교하십시오.

![레이어 정렬을 강제하기 위해 배리어가 있는 4-큐비트 양자 회로. 이제 레이어를 세기가 훨씬 쉬워졌습니다.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/compute-resources/aligned.avif)

이것들은 동일한 회로이며 동일한 수의 레이어를 가집니다. 그러나 두 번째에서는 정렬로 인해 회로에 다음이 있음을 쉽게 볼 수 있습니다:

* 두 개의 회전 레이어: Y 축을 중심으로 $\pi/5$만큼 회전하는 하나, Z 축을 중심으로 $\pi/4$만큼 회전하는 하나.
* 세 개의 얽힘 레이어. CNOT을 논리적 연산을 변경하지 않고는 병렬로 재순서할 수 없으므로 각 CNOT을 자체적인 "레이어"라고 부를 수 있습니다.
* 두 개의 추가 회전 레이어: Y 축을 중심으로 $\pi/3$만큼 회전하는 하나, Z 축을 중심으로 $\pi/2$만큼 회전하는 하나.
* 두 개의 추가 얽힘 레이어. 이번에는 첫 번째 레이어가 첫 번째 얽힘 레이어 세트보다 약간 더 병렬화되었음을 주목하십시오.

각 회로의 깊이는 9입니다.

### 2.3 측정 단위

양자 컴퓨팅에서 양자 시스템의 기능은 일반적으로 세 가지 주요 성능 지표인 규모, 품질 및 속도를 사용하여 평가됩니다. 이러한 지표는 양자 장치의 계산 잠재력을 설명할 뿐만 아니라 실제 응용 프로그램에서 리소스가 어떻게 관리되고 스케줄링되는지에 대한 정보를 제공합니다.

* 규모는 시스템의 양자 비트 (큐비트) 수를 나타내며, 장치가 보유할 수 있는 양자 정보의 양을 나타냅니다. 리소스 관리에서 이는 회로 폭 (주어진 양자 작업을 실행하는 데 필요한 큐비트 수) 에 직접적인 영향을 미칩니다. 양자 유닛은 할당된 작업을 지원하기에 충분한 큐비트를 가져야 합니다.

* 품질은 양자 연산이 얼마나 정확하게 수행되는지를 설명합니다. 이는 종종 레이어 충실도 (layer fidelity) 로 정량화되며, 이는 모든 큐비트에 걸쳐 양자 게이트의 전체 레이어를 실행하는 정확도를 측정합니다. 스케줄링 관점에서 더 높은 충실도는 더 깊은 회로를 안정적으로 실행할 수 있게 하여 오류 완화 또는 작업 분해의 필요성에 영향을 미칩니다.

* 속도는 CLOPS (Circuit Layer Operations Per Second) 로 측정되며, 시스템이 초당 실행할 수 있는 양자 연산 레이어 수를 나타냅니다. 이는 작업 실행의 처리량과 지연 시간에 영향을 미치며, 양자 유닛이 주어진 워크로드를 얼마나 빠르게 완료할 수 있는지를 결정하는 데 도움이 됩니다. 이 속도는 양자 컴퓨터에서 특히 중요한데, 큐비트가 클래식 대응물보다 노이즈와 오류의 영향을 훨씬 더 크게 받기 때문입니다. 큐비트가 양자 정보를 유용한 방식으로 유지할 수 있는 시간은 **결맞음 시간** (coherence time) 으로 설명되며, Heron r3 프로세서의 경우 일반적으로 200-300 $\mu\text{s}$ 정도입니다.

### 2.4 양자 지표와 클래식 지표 간의 차이

CLOPS를 FLOPS의 느슨한 양자 유사체로 생각할 수 있지만 몇 가지 주요 차이점이 있습니다. CLOPS는 양자 프로세서가 양자 회로를 실행할 수 있는 속도를 측정하며, 구체적으로 회로 내의 연산 레이어를 포함하며 회로 실행에 필요한 양자 및 필요한 클래식 계산을 모두 포함합니다. 이는 클래식 프로세서의 부동 소수점 산술 용량만 순수하게 측정하는 FLOPS와 달리, 양자 실행 시간과 회로 업데이트에 필요한 실시간 클래식 처리를 포함하는 양자 컴퓨터의 실행 속도에 대한 포괄적인 척도로서 IBM Quantum에 의해 개발되었습니다.

CLOPS는 기존 하드웨어에서 벤치마크할 수 있는 측정 가능한 성능 지표를 제공합니다. IBM Quantum은 CLOPS를 사용하여 다양한 양자 프로세서를 벤치마크했으며, 값은 IBM Quantum Platform의 [Compute resources](/computers) 페이지에서 찾을 수 있습니다. CLOPS 값은 하드웨어 기능, 게이트 속도, 클래식 처리 속도 및 이들의 통합에 따라 달라집니다.

큐비트 수는 주어진 QPU에 대해 고정된 숫자입니다. CLOPS와 품질은 정기적인 보정 및 유지보수에 따라 달라지며, 단일 QPU에서도 시간이 지남에 따라 약간 변할 수 있습니다.

함께, 이러한 지표는 양자 시스템이 어떻게 할당되고 스케줄링되는지를 안내합니다. 많은 경우 전체 양자 시스템은 단일 유닛으로 취급됩니다. 그러나 작업이 큐비트 수, 회로 깊이 또는 실행 속도 측면에서 하나의 유닛의 용량을 초과하는 경우, 회로 절단/니팅 (circuit cutting/knitting) 과 같은 기술을 사용할 수 있습니다. 회로 절단은 큰 양자 작업을 하드웨어 제한에도 불구하고 확장 가능한 양자 계산을 가능하게 하는 여러 양자 에 분배될 수 있는 더 작고 관리 가능한 하위 작업으로 분해하는 과정입니다. 회로 니팅은 회로 절단 후에 오는 과정을 의미합니다 — 더 작은 하위 회로의 결과를 다시 함께 "니팅"하거나 결합하는 클래식 후처리 단계입니다.

양자 컴퓨터는 RAM 또는 GPU 메모리와 같은 영구적이고 주소 지정 가능한 저장소라는 의미에서 전통적인 메모리를 가지고 있지 않습니다. 클래식 컴퓨팅 리소스는 메모리에 저장된 이산 비트를 가지며, 이를 통해 데이터를 계산 중에 저장하고 가져오고 재사용할 수 있습니다. 양자 리소스는 클래식 의미에서 메모리를 저장하지 않는 큐비트를 사용합니다. 대신 큐비트는 0과 1의 중첩을 동시에 나타내는 양자 상태에 존재하며, 상태 공간에서 지수적 병렬 처리를 가능하게 합니다. 그러나 큐비트 상태는 깨지기 쉽고 양자 상태를 괴시키지 않고는 중간 단계에서 복제하거나 결정론적으로 읽을 수 없으므로, 계산 중 영구적인 메모리 유사 동작은 존재하지 않습니다. 큐비트는 실행 전반에 걸쳐 일관된 상태로 유지되어야 하며, "메모리"는 본질적으로 양자 상태 자체입니다. 클래식 메모리는 내부 양자 메모리가 아닌 양자 프로세서와 함께 사용할 수만 있습니다. 이는 중요한 의미를 가집니다: 클래식 컴퓨팅 리소스는 중간 결과를 자유롭게 재사용하고 저장할 수 있지만, 양자 리소스는 계산을 방해하는 측정 없이는 이를 수행할 수 없습니다.

### 2.5 클래식 인프라와의 연결성

QPU는 소프트웨어 개발자가 QPU와 프로그래밍 방식으로 상호 작용할 수 있도록 하는 네트워크 및 다양한 애플리케이션 프로그래밍 인터페이스 (API) 를 통해 클래식 인프라에 연결될 수 있습니다. 이러한 API는 일반적으로 소프트웨어 개발 키트 (SDK) 및 라이브러리 (Qiskit와 같은) 뒤에 숨겨져 있으며, 프로그래밍 추상화 (3 장: 프로그래밍 모델에서 다룰 Qiskit Primitives와 같은) 의 형태로 전산 과학자에게 노출됩니다.

양자 및 클래식 리소스의 긴밀한 통합과 느슨한 통합을 구분하는 것이 가치가 있습니다. 현재 QPU는 클래식 컴퓨팅 리소스와 동일한 노드에 있지 않습니다. 실제로 QPU는 현재 PCIe를 통해 연결되지 않고 네트워크를 통해 연결됩니다. 이는 미래에 변할 수 있지만, QPU와 클래식 컴퓨팅 리소스에 최적의 환경 조건과 관련된 공학적 과제가 있습니다.

### 2.6 양자 리소스의 확장

양자 리소스의 확장은 수직 및 수평으로 분류될 수도 있습니다.

* 수직 확장은 칩당 큐비트 수를 늘리거나 장치의 충실도를 향상시키는 것입니다.
* 수평 확장은 칩을 플러 또는 클래식 인터커넥트로 연결하는 것입니다.

> (a) 양자 비트 또는 큐비트 - 0 또는 1 상태만 채택할 수 있는 클래식 대응물과 달리 0과 1의 중첩에 동시에 있을 수 있는 정보 단위.
(b) 초당 회로 레이어 연산 또는 CLOPS - 회로에서 파라미터를 로드하는 것과 같이 클래식 컴퓨팅 리소스와의 일부 인터페이싱을 포함하여 QPU가 초당 수행할 수 있는 순차 연산 수.



## 3. 리소스 관리

HPC 및 양자 리소스 모두 매우 귀중하고 복잡하므로 신중하게 관리해야 합니다. 이 섹션에서는 사용자 프로그램을 위한 리소스 관리 방법을 설명합니다. 컴퓨팅 인프라에서의 리소스 관리란 CPU, 메모리, 스토리지, 네트워크 대역폭과 같은 컴퓨팅 리소스의 사용을 (1) 계획하고, (2) 할당하고, (3) 제어/관리하여 효율적이고 효과적인 리소스 활용을 보장하는 과정을 의미합니다.

### 3.1 계획 - 리소스 추정

모든 프로그램은 리소스를 소모하며, 필요한 리소스를 추정하는 것은 효율적인 리소스 관리에 매우 중요합니다. 여기에는 프로그램을 실행하는 데 필요한 CPU, 메모리 및 기타 리소스의 양을 추정하는 것이 포함됩니다. 양자 리소스에도 마찬가지입니다. 그러나 양자 리소스는 완전히 차원이 다른 규모로 존재합니다. 일반적인 노트북에 있는 수십억 개의 클래식 비트와 비교하여, IBM Quantum® Heron r3 양자 프로세서는 156개의 큐비트를 가지고 있습니다. 시간과 비용도 고려해야 할 요소입니다. 현재 IBM Quantum은 사용자에게 월 10분의 QPU 시간을 할애하여 양자 컴퓨팅을 탐색할 수 있게 해주는 무료 플랜인 [Open Plan](https://www.ibm.com/quantum/products#access-plans)을 제공하고 있습니다. 일부 연구 기관은 너무 많은 QPU 시간이 필요하여 구내에 전용 IBM 양자 컴퓨터를 보유하고 있기도 합니다.

양자 컴퓨팅에 고유한 리소스 추정의 한 단계는 회로 깊이(circuit depth)입니다. 앞서 언급했듯이, 각 양자 게이트와 연산 사이의 각 지연 시간은 노이즈와 특정 확률의 오류를 수반합니다. 양자 회로의 깊이가 깊어질수록 노이즈는 커집니다. 여기에는 두 가지 미묘한 점이 있습니다. 첫째, 2-큐비트 게이트는 단일 큐비트 게이트보다 오류율이 훨씬 높기 때문에 단일 큐비트 깊이는 종종 무시할 수 있습니다. 둘째, 양자 칩의 모든 큐비트가 직접 연결되어 있는 것은 아닙니다. 필요한 얽힘을 수행하기 위해 정보를 큐비트에서 큐비트로 스왑해야 하는 경우가 있으며, 이 스왑 과정 자체에도 2-큐비트 게이트가 필요합니다. 이 스왑은 "트랜스파일레이션(transpilation)"이라고 불리는 과정에서 처리되는데, 이는 다른 목적에도 사용되는 복잡한 과정이며 다음 레슨에서 더 자세히 다룹니다. 따라서 관련 제한 값은 트랜스파일된 2-큐비트 깊이입니다. 높은 충실도(fidelity)의 결과를 얻을 수 있는 정확한 최대 깊이는 회로에 따라 다릅니다. 하지만 최신 오류 완화 기술을 활용하면 트랜스파일된 2-큐비트 깊이가 80 이상인 경우에도 높은 충실도의 결과를 얻을 수 있습니다.

### 3.2 할당 - 스케줄링

스케줄링은 프로그램에 리소스를 할당하고 실행을 관리하는 과정입니다. 여기에는 다음이 포함됩니다.

* 작업 제출: 사용자가 실행에 필요한 계산 작업과 리소스를 지정하여 HPC 시스템에 요청(작업)을 보내는 과정입니다.
* 리소스 할당: 제출된 작업의 요구 사항에 따라 사용 가능한 HPC 시스템 리소스(노드, CPU, 메모리 등)를 할당하는 것입니다.
* 작업 실행: 할당된 HPC 리소스에서 작업으로 정의된 계산 작업을 실제로 실행하는 것입니다.

양자 컴퓨터에도 이 모든 과정에 상응하는 것들이 있습니다.

* 작업은 사용자가 Qiskit Runtime을 활용하여 제출하며, 일반적으로 `Sampler`, `Estimator` 등과 같은 Qiskit Runtime 프리미티브를 사용합니다.
* 사용자는 접근 권한이 있는 백엔드 목록에서 선택합니다. 사용 가능한 백엔드의 전체 목록은 IBM Quantum Platform의 [Compute resources](/computers) 페이지에서 확인할 수 있습니다. 일반적으로 가장 한가한 양자 컴퓨터를 사용하는 것이 일반적입니다. 하지만 디바이스 레이아웃 고려 사항, 이전 계산의 재현 등의 이유로 특정 백엔드를 사용하는 것이 중요할 수 있는 경우도 있습니다.
* 양자 작업의 실행은 HPC의 경우와 유사합니다. 이미 몇 가지 차이점이 개략적으로 설명되었지만, 여기서 다시 언급할 가치가 있는 몇 가지가 있습니다. 현재 QPU는 일반적으로 클래식 컴퓨팅 리소스와 동일한 노드에 위치하지 않으며 네트워크를 통해 연결됩니다. 이는 스케줄링에 영향을 미칠 수 있습니다. 또한 양자 컴퓨터는 상당한 대기열 시간을 가질 수 있으며, 이 대기열 시간은 변동이 커서 정확한 타이밍 제어가 어렵습니다. 전용 시스템의 경우 이 상황이 다를 수 있으며, 이는 양자 컴퓨터의 내부 관리 방식에 따라 다릅니다.

### 3.3 제어/관리 - 워크로드 관리

워크로드 관리(오케스트레이션이라고도 함)는 여러 프로그램과 해당 리소스 요구 사항을 관리하는 과정입니다. 여기에는 다음이 포함됩니다.

* 리소스 프로비저닝: 하드웨어 및 소프트웨어 설정을 포함하여 HPC 리소스를 준비하고 작업에서 사용할 수 있도록 준비시키는 과정입니다. 나중에 살펴보겠지만, QPU는 이전 섹션의 주의 사항을 전제로 클래식 HPC 리소스와 유사하게 프로비저닝할 수 있는 컴퓨팅 리소스입니다.
* 작업 스케줄링: 스케줄러 소프트웨어가 어떤 작업을 언제, 어떤 리소스에서 실행할지 결정하고, 우선 순위와 대기열을 관리하여 HPC 시스템을 효율적으로 활용하는 활동입니다. 이 광범위한 진술은 양자 리소스에도 적용되지만, 다른 리소스에 비해 타이밍에 대한 제어 권한이 적을 수 있습니다.

![워크로드(상자로 표시됨)가 시간 축과 리소스 축을 가진 2차원 그리드에 최적화되도록 구성 및 배열되는 것을 보여주는 이미지](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/compute-resources/resource-manage.svg)

### 3.4 예시

리소스 관리를 이해하기 위한 맥락으로 잘 알려진 작업을 고려해 보겠습니다. 바로 큰 수의 소인수를 찾는 것입니다. 여기서 사용되는 알고리즘이 모든 잠재적 약수를 무작정 대입하여 확인하는 브루트 포스 방식에 의존한다고 가정해 보겠습니다. 이것이 항상 가장 효율적인 방법은 아니지만, 워크로드가 어떻게 관리될 수 있는지 쉽게 이해할 수 있습니다.

계획 - 리소스 추정

* 소인수 분해에 얼마나 많은 CPU 시간과 메모리가 필요할지 추정합니다.
* 작업의 병렬화를 계획합니다. 몇 개의 CPU/코어를 사용할 것인가요?

할당 - 스케줄링

* 작업 제출 시, 스케줄러는 소인수 분해 작업에 CPU 코어와 메모리를 할당합니다. 예를 들어, 4개의 코어 각각에 끝자리가 `1, 3, 7, 9`로 끝나는 모든 잠재적 약수를 각각 할당할 수 있습니다.
* 작업 실행: 소인수 분해 알고리즘이 실행되어 할당된 리소스에서 나눗셈이나 기타 분해 단계를 수행하며 작업이 완료될 때까지 진행됩니다.

제어/관리 - 워크로드 관리

* 시스템은 처리량을 최적화하기 위해 소인수 분해 작업의 순서와 타이밍을 오케스트레이션합니다.
* 가장 쉽게 상상할 수 있는 경우는 코어 중 하나가 목표 소인수를 찾는 것입니다. 이 경우 다른 코어의 계산을 중지하여 다음 작업에 사용할 수 있도록 해야 합니다.

고성능 컴퓨팅 환경은 이러한 단계를 수행하고 리소스를 관리하기 위해 특수한 소프트웨어를 사용합니다. 다음 섹션에서는 널리 사용되는 리소스 관리 소프트웨어 시스템인 Slurm에 대해 알아보겠습니다.

### 3.5 양자 리소스를 사용한 예시

이 과정의 다른 레슨에서 다룰 워크로우는 샘플 기반 양자 대각화(SQD)를 사용하여 화학적 바닥 상태와 에너지를 결정하는 것입니다. 이에 대한 자세한 내용은 레슨 4에서 다루며, [IBM Quantum Learning](https://quantum.cloud.ibm.com/learning/)에서 [SQD에 대한 이 과정](https://quantum.cloud.ibm.com/learning/courses/quantum-diagonalization-algorithms) 및 관련 방법을 방문할 수도 있습니다. 이 논의에서 우리가 알아야 할 모든 것은 워크로우드가 다음을 포함한다는 것입니다:

* 양자 회로 준비
* 양자 회로 측정
* 측정 결과를 사용하여 문제를 유용한 부분 공간으로 투영
* 클래식 컴퓨팅 리소스를 사용하여 더 작고 투영된 행렬을 대각화
* 전하 보존과 같은 고려 사항을 통해 자체 일관성을 보장하기 위한 반복, 또는 변분 매개변수가 있는 경우 양자 회로의 가능한 반복 작업

계획 - 리소스 추정

* 전자 오비탈을 큐비트에 매핑하여 필요한 큐비트의 수를 설정합니다.
* 시스템의 매핑된 해밀토니안과 (변분일 수 있는) 상태를 양자 회로로 결합하고, 트랜스파일된 2-큐비트 깊이를 확인합니다. 이것이 합리적인지 확인합니다.
* 투영할 부분 공간의 크기를 추정합니다. 이를 통해 대각화에 얼마나 많은 CPU 시간과 메모리가 필요할지 추정합니다.
* 작업의 병렬화를 계획합니다. 몇 개의 CPU/코어를 사용할 것인가요?

할당 - 스케줄링

* 사용자가 QPU를 선택합니다. 트랜스파일레이션 과정은 추상 양자 회로의 큐비트를 QPU의 물리적 큐비트에 자동으로 매핑합니다. 이는 추상 회로가 칩에 존재하지 않는 직접 연결을 가정할 수 있기 때문에 등 여러 이유로 중요합니다.
* Qiskit Runtime을 통해 작업이 제출되면, 선택된 QPU의 대기열에 진입합니다. 사용자는 대기열 시간을 제어할 수 없으나, 전용 시스템의 경우 이는 다를 수 있습니다.
* 클래식 컴퓨팅 리소스는 양자 결과를 기다립니다.
* 대각화 작업이 HPC 리소스에 제출됩니다. 작업 제출 시, 스케줄러는 대각화 작업에 CPU 코어와 메모리를 할당합니다.
* 작업 실행: 대각화 알고리즘이 실행되어 작업이 완료될 때까지 더 작게 투영된 행렬을 대각화합니다.

제어/관리 - 워크로드 관리

* 시스템은 전체적으로 양자 및 클래식 단계의 순서와 타이밍을 오케스트레이션합니다. 예를 들어, 투영된 행렬이 대각화되어 바닥 상태 에너지를 얻은 후, 수렴 기준에 따라 워크플로우가 새 양자 회로(새 변분 매개변수 포함)로 루프백 할 수 있습니다.
* 바닥 상태 에너지가 수렴 기준을 충족하면 모든 코어의 계산이 중지됩니다.

고성능 컴퓨팅 환경은 이러한 단계를 수행하고 리소스를 관리하기 위해 특수한 소프트웨어를 사용합니다. 다음 섹션에서는 널리 사용되는 리소스 관리 소프트웨어 시스템인 Slurm에 대해 알아보겠습니다. Slurm이 위에서 설명한 모든 단계에 대한 도구를 갖추고 있는 것은 아니라는 점에 유의하는 것이 중요합니다. Slurm은 작업 계획에 대한 지원을 제공하지 않으며, 워크로드 구성 요소 간의 통신과 같은 세부적인 워크로드 관리도 지원하지 않습니다. 이는 QPU가 일반적으로 네트워크를 통해 접근되기 때문에 HPC 내 양자 컴퓨팅의 현재 상태에 잘 부합합니다.

## 4. 소프트웨어: Slurm

이번 섹션에서는 이번 장에서 배운 개념을 인기 있는 리소스 관리 시스템인 Slurm을 사용하는 실습에 적용해 보겠습니다.

### 4.1 Slurm 소개

Slurm은 고성능 컴퓨팅 환경에서 널리 사용되는 오픈소스 리소스 관리 시스템입니다. 리소스 관리, 작업 스케줄링, 시스템 성능 모니터링을 위한 포괄적인 도구 세트를 제공합니다.

다음 내용을 포함하여 Slurm 사용의 기본 사항을 다룹니다:

* 작업 제출
* 리소스 할당
* 작업 모니터링

본 과정의 각 학생에게 HPC 리소스를 제공하는 것은 매우 어렵기 때문에, 약간의 편법을 써서 Slurm을 사용하여 실제 HPC 클러스터를 모방하되 소규모로 구현된 Docker 이미지가 포함된 저장소를 제공할 것입니다. 이는 안전하고 재현 가능한 환경에서 학습한 개념을 실습하는 데 도움이 될 것입니다.

현재 모든 양자 및 고전 리소스는 전체 실험 기간 동안 할당된다는 점에 유의해 주시기 바랍니다. 현재 혼합 리소스의 교차 할당은 지원되지 않습니다. 마지막으로 주의하실 점은, 작업이 시작된 후에도 숙련된 HPC 사용자가 기대하는 것처럼 양자 시스템이 직접 제어되지는 않는다는 것입니다. 작업은 임의의 x86 노드에서 실행되는데, 이 노드에서 Qiskit Runtime 서비스가 구동되고, 이 런타임 서비스는 사용자가 직접 제어할 수 없는 또 다른 스케줄러에 연결됩니다. 이러한 워크플로우와 관련된 이슈는 GPU 노드에 대한 독점적 접근을 추구했던 초기 경험을 가진 HPC 사용자들에게는 익숙할 수 있습니다(`gres`의 초기 사용 사례).

### 4.2 설치 지침 및 설정 개요

양자 및 HPC 리소스를 결합하는 실습을 위해서는 실제 HPC 환경에 액세스할 수 있거나 로컬 머신에서 HPC 환경을 시뮬레이션해야 합니다. Docker를 사용한 로컬 설정을 위한 설치 가이드는 [이 저장소](https://github.com/christopherporter1/hpc-course-demos)에서 확인할 수 있습니다. 이 가이드는 IBM Cloud® 계정 설정 및 QRMI용 SPANK 플러그인 설치를 지원하는 링크를 포함하고 있습니다. 또한 해당 저장소에는 환경을 테스트할 수 있는 여러 Python 파일이 포함되어 있습니다.

설치를 완료한 후에는 아래 명령어를 사용하여 터미널에서 Slurm의 컴퓨팅 리소스를 확인해 보겠습니다. 설치가 성공적으로 이루어졌다면 총 3개의 가상 노드가 확인되어야 합니다.

quantum*    up   infinite     1  idle q1

```shell

저희는 일반 파티션과 양자 파티션, 두 개의 노드 그룹을 운영합니다. 일반 파티션은 고전적 리소스에만 접근할 수 있는 노드들로 구성되고, 양자 파티션은 양자 리소스에만 접근할 수 있습니다. 각 노드의 자세한 정보는 `scontrol show nodes` 명령어를 통해 확인할 수 있습니다.

~
```

이제 리소스 관리자에게 이 프로그램을 실행하는 데 필요한 리소스를 알려줘야 합니다. Slurm은 제출 스크립트를 통해 작업에 필요한 모든 메타데이터를 지정하는 방법을 제공합니다. 제출 스크립트는 Slurm 관련 어노테이션이 포함된 셸 스크립트입니다. 이러한 어노테이션을 사용하면 리소스 요구 사항, 스케줄 매개변수 등을 지정할 수 있습니다. 이를 위해 `hello_world.sh`라는 셸 스크립트를 만들어 보겠습니다.

```shell
$ vim hello_world.sh


#SBATCH --job-name=hello-world
#SBATCH --output=hello-world.out
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1
#SBATCH --partition=normal

srun hello_world.py
~

```

제출 파일을 살펴보고 여기서 무슨 일이 일어나고 있는지 살펴보겠습니다.

`#SBATCH` 지시문은 프로그램 실행에 필요한 요구 사항을 알려주는 특정 주석입니다. 여기서는 리소스 양(노드 수, 노드당 작업 수, 노드 및 작업당 작업 및 CPU 수)과 출력 파일 이름과 같은 기타 옵션을 지정할 수 있습니다. 전체 옵션 목록은 Slurm에 대한 [문서](https://slurm.schedmd.com/documentation.html)에서 확인할 수 있습니다.

이제 Slurm 작업을 실행할 차례입니다. 'sbatch'는 제출 파일을 수락하고 Slurm에서 실행할 작업을 대기열에 추가하는 명령입니다.

```shell

```

작업이 완료되면 출력 파일을 확인하여 결과를 확인할 수 있습니다.

### 4.4 Slurm에서 간단한 Qiskit 헬로 월드 예제 실행하기

다음으로, 양자 리소스를 사용해 보겠습니다. 양자 리소스를 사용하는 간단한 "Hello, Qiskit" 프로그램을 만들고 실행해 보겠습니다.

```shell
$ vim hello_qiskit.py
```


```python

# hello_qiskit.py
from qiskit import QuantumCircuit
from qiskit.quantum_info import SparsePauliOp
from qiskit.transpiler import generate_preset_pass_manager
from qiskit_ibm_runtime import EstimatorV2 as Estimator

# Create a new circuit with two qubits
qc = QuantumCircuit(2)

# Add a Hadamard gate to qubit 0
qc.h(0)

# Perform a controlled-X gate on qubit 1, controlled by qubit 0
qc.cx(0, 1)

observables_labels = ["IZ", "IX", "ZI", "XI", "ZZ", "XX"]
observables = [SparsePauliOp(label) for label in observables_labels]

# switch to QRMI service
from qiskit_ibm_runtime import QiskitRuntimeService

service = QiskitRuntimeService()

backend = service.backend("...")

# Convert to an ISA circuit and layout-mapped observables.
pm = generate_preset_pass_manager(backend=backend, optimization_level=1)
isa_circuit = pm.run(qc)

# Construct the Estimator instance.

estimator = Estimator(mode=backend)
estimator.options.resilience_level = 1
estimator.options.default_shots = 5000

mapped_observables = [
    observable.apply_layout(isa_circuit.layout) for observable in observables
]

# One pub, with one circuit to run against five different observables.
job = estimator.run([(isa_circuit, mapped_observables)])

job_result = job.result()

pub_result = job.result()[0]

print("Result", pub_result)

```

여기서는 양자 자원 관리 인터페이스(QRMI)를 사용합니다. QRMI는 IBM, Pasqal, 하트리 센터, 그리고 라지피그 공과대학(RPI)이 공동 개발한 양자 자원 및 작업 지원을 위한 Slurm SPANK 플러그인입니다. 우리는 무작위 초기값과 간단한 관측량을 가진 간단한 Pauli-2 설계 회로를 만들었고, Estimator를 사용하여 기대값을 얻기 위해 이를 실행할 것입니다. 이를 실행하려면 양자 자원을 필수 요구 사항으로 하는 제출 스크립트 `hello_qiskit.sh`가 필요합니다.

```shell
$ vim hello_qiskit.sh

#SBATCH --job-name=hello-qiskit
#SBATCH --output=hello_qiskit.out
#SBATCH --nodes=1
#SBATCH --ntasks-per-nodes=1
#SBATCH --cpus-per-task=1
#SBATCH --partition=quantum
#SBATCH --gres=qpu:1

srun python /data/ch2/hello_qiskit/hello_qiskit.py
~
```

제출 파일을 가져와서 어떤 내용이 있는지 살펴보겠습니다. `gres`라는 새로운 옵션이 하나 추가되었습니다. `gres`는 Slurm에서 추가적인 연산 리소스를 정의하는 옵션입니다. 이 경우 이 새로운 리소스는 양자 컴퓨팅 리소스가 될 것입니다. 리소스와 양자 리소스를 사용할 수 있는 클러스터 파티션을 지정했으므로, Qiskit 프리미티브는 할당된 리소스를 사용하여 양자 페이로드를 실행합니다.

이제 Slurm 작업을 실행해 보겠습니다.


```shell
$ sbatch hello_qiskit.sh
```

그럼 이제 `squeue` 명령어를 사용하여 프로그램의 상태를 확인해 보겠습니다.

```shell
$ squeue
#             JOBID PARTITION     NAME         USER ST       TIME  NODES NODELIST(REASON)
#                 1 main          hello_qiskit  root R        0:01      1 q1
```

작업이 완료된 후 로그와 결과를 확인할 수 있습니다.

```shell
$ cat hello_qiskit.out | grep Exp
Expectation Value: 0.8372900070983516
```
