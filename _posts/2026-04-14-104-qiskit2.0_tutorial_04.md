### 2. 실행 모드

#### 1. 세 가지 실행 모드

IBM Quantum 은 양자 프로그램을 실행하기 위해 세 가지 구별되는 실행 모드를 제공합니다:

1.  Job Mode  ( `mode=backend` )
   - 컨텍스트 매니저 없이 Sampler 또는 Estimator 의 단일 요청.
   - 일회성 실행을 위한 가장 간단한 모드.
   - primitive 를 초기화할 때  `mode=backend`  를 지정합니다. 

2.  Batch Mode  ( `mode=batch` )
   - 서로 조건적 관계가 없는 독립적으로 실행 가능한 작업으로 구성된 다중 작업 워크로드.
   - 작업은 효율성을 위해 함께 스케줄링됩니다.
   - 제출 순서대로 실행된다는 보장은 없으며, 다른 사용자의 작업과 병렬로 실행될 수 있으며, 배치된 작업 사이에 QPU 보정 작업도 실행될 수 있습니다.
   - 작업에서 제출된 첫 번째 작업에 대한 대기열 시간이 감소하지 않습니다.
   - primitive 를 초기화할 때  `mode=batch`  를 지정합니다.

3.  Session Mode  ( `mode=session` )
   - 다중 작업 워크로드를 실행하기 위한 전용 창.
   - QPU 에 대한 독점적 접근 권한이며 보정 작업을 포함하여 다른 작업은 실행될 수 없습니다.
   - 고전적 자원과 양자 자원 간의 빈번한 통신이 필요한 반복적 작업에 특히 유용합니다.
   - 작업에서 제출된 첫 번째 작업에 대한 대기열 시간이 감소하지 않습니다.
   - primitive 를 초기화할 때  `mode=session`  를 지정합니다.
   - 배치는 "함께 예약만 한 효율화" 인 반면, 세션은 "완전 혼자 쓰는 전용 구간" 입니다.

### 3. 실행 모드 선택

#### 1. 각 모드를 사용할 시기

| 모드 | 최적의 용도 | 주요 기능 |
|------|----------|-------------|
|  Batch  | 독립적인 회로의 거대 컬렉션 | • 전체 배치가 함께 스케줄링됨 • 고전적 계산이 병렬로 실행됨 • 작업 간 지연 시간 최소화 • 유연함 - 필요시 나머지 작업 취소 가능 • 세션보다 비용이 덜 듬 |
|  Session  | 변분 알고리즘, 반복적 워크로드 | • 모든 배치 모드 기능 • QPU 에 대한 전용 접근 권한 • 입력이 점진적으로 도착할 때 유용 • 낮은 지연 시간이 필요한 워크로드에 더 좋음 |
|  Job  | 작은 실험, 빠른 테스트 | • 가장 간단한 인터페이스 • 배치보다 더 빨리 실행될 수 있음 • 학습 및 테스트에 좋음 |

#### 2. 모범 사례

- 기본적으로 배치 모드를 사용하십시오. 특히 모든 입력이 준비된 경우 세션보다 비용 효율적입니다.

- 배치 모드는 총 실행 시간을 줄이기 위해 여러 primitive 작업을 한 번에 제출하는 데 이상적입니다.

- 세션 모드는 반복적 워크로드 또는 전용 QPU 접근 권한이 필요할 때 가장 좋지만 일반적으로 더 비쌉니다.

- Job 모드는 단일 primitive 요청에만 사용해야 합니다.

- Open Plan 사용자는 세션을 사용할 수 없습니다.

#### 3. 권장 사항:

- 각 작업에는 고정 오버헤드가 있습니다. 작업이 1 분 미만의 QPU 시간을 사용하는 경우 여러 작업을 단일 작업으로 결합하십시오.

- 작업이 1 분 이상 걸리거나 결합할 수 없는 경우, QPU 가 한 번에 하나의 작업만 실행하더라도 병렬 고전적 처리를 활용하기 위해 (최대 5 개 작업) 배치 또는 세션 모드에서 여러 작업을 제출하십시오.

- 특히 세션에서 워크로드 크기 및 병렬성을 조정하여 wall-clock 시간(총시간) 및 비용을 최소화하십시오; 큰 작업을 분할하는 것이 더 효율적일 수 있습니다.

### 4. 배치에서 작업 실행

#### 1. 배치 실행 기본 사항

배치 모드를 사용하면 함께 스케줄링되는 여러 양자 작업을 제출할 수 있습니다. 이는 개별 작업을 별도로 제출하는 것보다 더 효율적입니다.

```python
# if your account is already saved, this will load it
service = QiskitRuntimeService()
```

#### 2. 배치 열기

`Batch`  클래스를 초기화하거나 컨텍스트 매니저  `with Batch()`  를 사용하여 수행할 수 있습니다.

```python
# Method 1: Explicit batch creation and closure
backend = service.least_busy(operational=True, simulator=False)

#Create a batch execution context
batch = Batch(backend=backend, max_time="5m")

# Configure primitives to use batch mode
estimator = Estimator(mode=batch)
sampler = Sampler(mode=batch)

# Close the batch explicitly (no context manager used)
batch.close()
```

```python
# Method 2: Using context manager
backend = service.least_busy(operational=True, simulator=False)
print(f"Selected backend: {backend.name}")
# Automatically handle batch lifecycle
with Batch(backend=backend, max_time="5m"):
    estimator = Estimator()  
    sampler = Sampler()

#Batch automatically closes when context exits
```


배치에서 실행할 회로 준비

```python
# Create example circuits and observables
qc1 = QuantumCircuit(2)
qc1.h(0)
qc1.cx(0, 1)
qc1.measure_all()

qc2 = QuantumCircuit(2)
qc2.h(0)
qc2.cx(0, 1)
qc2.rz(0.5, 1)
qc2.measure_all()

# Create observables for estimator
observable1 = SparsePauliOp("Z"*156)#ibm_kingston기준(156-qubit)
observable2 = SparsePauliOp("X"*156)


pm = generate_preset_pass_manager(optimization_level=1, backend=backend)
isa_circuit1 = pm.run(qc1)
isa_circuit2 = pm.run(qc2)
# Create primitive units (pubs) for execution
estimator_pub = (isa_circuit1, observable1)
#의미: isa_circuit1이라는 회로를 실행해서, observable1(=Z 133개)의 기댓값을 계산하라
# Pub 구조: (회로, 관측가능량) 또는 (회로, 관측가능량, parameter_set) 등 가능

sampler_pub = isa_circuit2
#의미: isa_circuit2라는 회로를 실행해서 샘플(측정 결과 비트스트링) 을 가져와라
#Pub 구조: (회로) 또는 (회로, parameter_set) 등 가능
```

#### 3. 배치 지속 시간 제한

최대 TTL 은  `max_time`  으로 정의할 수 있으며, 작업 배치가 시작될 때 시작됩니다. 최대 시간에 도달하면 배치가 닫히고 실행 중인 작업은 완료되지만 대기열에 있는 모든 작업은 실패합니다.

| 플랜 유형 | 최대 배치 지속 시간 |
|-----------|-----------------------|
| Open Plan | 10 분 |
| Paid Plans | 8 시간 |

또한 구성할 수 없는 대화형 TTL 도 있습니다 (모든 플랜에 대해 1 분), 창 내에 대기열에 작업이 없으면 작업이 일시적으로 비활성화됩니다.

 참고:  이러한 제한은 배치 내 모든 작업의 총 실행 시간에 적용됩니다.

### 5. 배치 닫기

배치는 컨텍스트 매니저가 종료될 때 자동으로 닫힙니다. 그 후 배치는 새로운 작업을 받아 들이는 것을 중단하고 최대 TTL 에 도달할 때까지 대기열에 있거나 활성인 모든 작업의 실행을 완료한 후 영구적으로 닫힙니다.

컨텍스트 매니저를 사용하지 않는 경우 배치를 수동으로 닫아야 합니다. 배치를 열어두고 나중에 작업을 제출하면 시작 전에 최대 TTL 에 도달하여 취소될 위험이 있습니다.  `batch.close()`  를 호출하면 새로운 제출을 방지하면서 이미 제출된 작업이 완료되고 결과를 검색할 수 있습니다.

```python
with Batch(backend=backend) as batch:
    estimator = Estimator()
    sampler = Sampler()
    job1 = estimator.run([estimator_pub])
    job2 = sampler.run([sampler_pub])
 
# The batch is no longer accepting jobs but the submitted job will run to completion.
result = job1.result()
result2 = job2.result()
```

#### 1. 실제 예시: 회로 일괄 실행

```python
# Execute circuits using batch mode with context manager
with Batch(backend=backend) as batch:
    estimator = Estimator()
    sampler = Sampler()
    
    # Submit jobs to batch
    estimator_job = estimator.run([estimator_pub])
    sampler_job = sampler.run([sampler_pub])
 
# Results are available after batch context closes
# The batch is no longer accepting jobs but submitted jobs run to completion
estimator_result = estimator_job.result()
sampler_result = sampler_job.result()

print("Batch execution completed successfully!")
print(f"Estimator result: {estimator_result}")
print(f"Sampler result: {sampler_result}")
```

### 7. 대량의 회로 배치 처리

<span style="color:red" >  아래 셀을 실행하면 IBM 계정 크레딧에서 약 1 분이 사용됩니다, 회로 실행 시간을 줄이기 위해 회로 수를 줄일 수 있습니다. 

```python
# Create a large number of circuits
max_circuits_per_batch = 10  # Limit per batch submission
total_circuits = 50
circuits = [random_circuit(3, 3) for _ in range(total_circuits)]

# Partition circuits into manageable batches
all_partitioned_circuits = []
for i in range(0, len(circuits), max_circuits_per_batch):
    all_partitioned_circuits.append(circuits[i : i + max_circuits_per_batch])

jobs = []

# Submit partitioned batches
with Batch(backend=backend):
    sampler = Sampler()
    for i, partitioned_circuits in enumerate(all_partitioned_circuits):
        pm = generate_preset_pass_manager(optimization_level=1, backend=backend)
        isa_circuit = pm.run(partitioned_circuits)
        job = sampler.run(isa_circuit)
        jobs.append(job)
        print(f"Submitted batch {i+1} with {len(partitioned_circuits)} circuits")

print(f"\nCreated {total_circuits} circuits")
print(f"Partitioned into {len(all_partitioned_circuits)} batches of maximum {max_circuits_per_batch} circuits each")
print(f"Successfully submitted {len(jobs)} jobs to batch")
```

## 2. 실제 하드웨어에서 회로 실행 시연

**개요:** 이 노트북은 실제 하드웨어에서 회로를 실행하는 방법을 다룹니다.

* 실제 하드웨어에서의 실행 단계
* 프리미티브: 샘플러(Sampler), 에스티메이터(Estimator), 이들의 구현 및 입출력 구조
* 브로드캐스팅 규칙: 파라미터 세트와 관측가능량이 어떻게 효율적으로 기댓값을 계산하는 데 사용되는지
* 세션 실행: 세션을 열고, 닫고, 세부 정보를 보는 방법

```python
# Import necessary libraries
from qiskit import QuantumCircuit
from qiskit_ibm_runtime import QiskitRuntimeService
```

### 1. 대상 하드웨어에서 실행

실제 하드웨어에서 양자 회로를 실행하는 것은 다음과 같은 여러 핵심 단계를 포함합니다:

1. **인증**: IBM Quantum Runtime 서비스에 연결
2. **백엔드 선택**: 사용 가능한 양자 프로세서 선택
3. **회로 준비**: 특정 하드웨어 제약사항에 맞춰 회로를 트랜스파일링
4. **작업 제출**: 실행을 위해 회로 전송
5. **결과 조회**: 측정 결과 획득 및 분석

<span style="color:red"> **Running the cell below will use around 10-15 seconds from your IBM account credit**
</span>

```python
# Example:  Running a Quantum Circuit on Real Hardware

# --- STEP 1: Authentication ---
# if account is already saved, this will load it
service = QiskitRuntimeService()
# replace <instance_name> with your instance name
# replace <token> with your IBM Quantum API token
# Uncomment and use the following line if you need to specify instance and token
# srvice = QiskitRuntimeService(channel="ibm_cloud",instance="<instance_name>",token=<token>)

   
# --- STEP 2: Backend Selection ---
# Choose the least busy available backend to minimize queue time with real hardware only
backend = service.least_busy(operational=True, simulator=False)
print(f"Using backend: {backend.name}")
print(f"Backend properties: {backend.num_qubits} qubits")

# --- STEP 3: Circuit Preparation ---
# Create a simple Bell state circuit
qc = QuantumCircuit(2, 2)
qc.h(0)                   
qc.cx(0, 1) 
qc.measure([0, 1], [0, 1])

# Display the circuit
qc.draw('mpl')

# Transpile the circuit for the specific hardware backend
# This transpiles the circuit to the backend's: Native gate set,  Qubit connectivity and Optimization level (0-3, where 3 is most optimized)
pm = generate_preset_pass_manager(backend=backend, optimization_level=3)
transpiled_circuit = pm.run(qc)

# Define the observable we want to measure (ZZ correlation for example)
# ⟨ZZ⟩는 두 큐비트가 얼마나 "같은 방향으로 정렬되어 있는가"를 나타내는 척도
ZZ = SparsePauliOp.from_list([("ZZ", 1)])

# --- STEP 4: Job Submission ---
# Create an Estimator instance and run the job
estimator = Estimator(mode=backend)

# Map observables to the transpiled circuit layout
observables = [ZZ.apply_layout(transpiled_circuit.layout)]

# Submit the job
job = estimator.run([(transpiled_circuit, observables)])
print(f"\nJob ID: {job.job_id()}")
print("Job submitted successfully! Check the IBM Quantum dashboard for status.")
```

```python
# --- STEP 5: Result Retrieval ---
result = job.result()
print(f"Expectation value: {result[0].data.evs}")
```


### 2. 프리미티브 이해

프리미티브는 하드웨어 특화적인 세부사항을 추상화하는 기초적인 양자 컴퓨팅 연산입니다. 이들은 일반적인 양자 작업을 위한 표준화된 인터페이스를 제공합니다.

#### 1. 프리미티브:

1. **샘플러 (Sampler)**:
   - 용도: 양자 회로부터 측정 결과 샘플링
   - 출력: 카운트 또는 확률 분포

2. **에스티메이터 (Estimator)**:
   - 용도: 관측가능량의 기댓값 계산
   - 출력: 기댓값 ± 표준 오차

#### 2. 프리미티브 구현:

**기본 클래스:**
- `BaseSamplerV2`: 샘플러를 위한 기본 클래스, 모든 샘플러 구현체는 이 클래스를 상속합니다.
- `BaseEstimatorV2`: 에스티메이터를 위한 기본 클래스, 모든 에스티메이터 구현체는 이 클래스를 상속합니다.

**V2 프리미티브:**
- `SamplerV2`: 향상된 기능을 갖춘 최신 샘플러
- `EstimatorV2`: 향상된 성능과 기능을 갖춘 최신 에스티메이터

**대안:**
- `StateVectorSampler`: 시뮬레이터 기반 샘플러
- `StateVectorEstimator`: 상태 벡터 시뮬레이션을 사용하는 시뮬레이터 기반 에스티메이터
- `BackendSamplerV2`: 백엔드 특화 샘플러 최적화
- `BackendEstimatorV2`: 백엔드 특화 에스티메이터 최적화

### 3. 프리미티브 입력 및 출력

**입력**:

#### 1. 샘플러 입력 구조

각 샘플러 'Pub'(Primitive Unified Bloc)은 다음을 받아드립니다:
1. **양자 회로**: 실행할 회로 (샘플링될 큐비트에 대한 측정을 포함해야 함)
2. **파라미터**: 파라미터화된 회로를 위한 파라미터 값 (회로가 파라미터화된 경우에만 필요)
3. **샷 (Shots)**: 회로 실행 횟수 (선택사항)

#### 2. 에스티메이터 입력 구조

각 에스티메이터 'Pub'은 다음을 받아드립니다:
1. **양자 회로**: 실행할 회로 (대상 백엔드에 맞춰 트랜스파일링되어야 함)
2. **관측가능량**: 측정할 하나 이상의 관측가능량 (`Pauli`, `SparsePauliOp`, `PauliList` 또는 `str` 객체로 제공)
3. **파라미터**: 파라미터화된 회로를 위한 파라미터 값 (선택사항)
4. **정밀도**: 오류 완화를 위한 목표 정밀도 (선택사항)

<span style="color:red"> **Running the cell below will use around 5 minutes from your IBM account credit.**
</span>

```python
# Example 1: Using the Estimator Primitive with Multiple Parameters and Observables
# --- Create a parameterized circuit ---
circuit = QuantumCircuit(2)
circuit.h(0)                     
circuit.cx(0, 1)                 
circuit.ry(Parameter("a"), 0)    # Parameterized rotation Y
circuit.rz(Parameter("b"), 0)    # Parameterized rotation Z
circuit.cx(0, 1)                 
circuit.h(0)

# Display the parameterized circuit
print("Parameterized Circuit:")
circuit.draw('mpl')

# --- Transpile for hardware ---
pm = generate_preset_pass_manager(optimization_level=1, backend=backend)
transpiled_circuit = pm.run(circuit)
layout = transpiled_circuit.layout

# --- Define parameter sweep ---
# Create 100 different parameter combinations
# Shape: (100, 2) where 100 is number of parameter sets, 2 is parameters per set
params = np.vstack([
    np.linspace(-np.pi, np.pi, 100),        # Parameter 'a' values
    np.linspace(-4 * np.pi, 4 * np.pi, 100) # Parameter 'b' values
]).T

print(f"\nParameter array shape: {params.shape}")
print(f"First 5 parameter sets:\n{params[:5]}")

# --- Define multiple observables ---
# measure three different observables for each parameter set
# Shape: (3, 1) where 3 is number of observable sets, 1 is observables per set
observables = [
    [SparsePauliOp(["XX", "IY"], [0.5, 0.5])],   # Linear combination of XX and IY
    [SparsePauliOp("XX")],                       # XX
    [SparsePauliOp("IY")],                       # IY
]

# Apply the same layout as the transpiled circuit
observables = [
    [observable.apply_layout(layout) for observable in observable_set]
    for observable_set in observables
]

print(f"\nNumber of observable sets: {len(observables)}")
print(f"Observables per set: {len(observables[0])}")

# --- Create the Estimator Pub ---
# Total calculations: 1 circuit  × 3 observables × 100 parameter sets = 300 expectation values
estimator_pub = (transpiled_circuit, observables, params)

# --- Execute with Estimator ---
estimator = Estimator(mode=backend)
job = estimator.run([estimator_pub])

print(f"\nJob submitted! Job ID: {job.job_id()}")
```

```python
# --- Process Results ---
result = job.result()
print(f"\nResult shape: {result[0].data.evs.shape}")  # Should be (3, 100)
print(f"First expectation values: {result[0].data.evs[:, :5]}")
```


#### 3. 브로드캐스팅 규칙

:Qiskit Estimator V2의 broadcasting은 기본적으로:
“리스트 길이가 같으면 zip, 다르면 broadcast”

브로드캐스팅을 사용하면 단일 프리미티브 호출로 여러 파라미터 세트와 관측가능량에 대한 기댓값을 효율적으로 계산할 수 있습니다. 이 규칙은 NumPy 브로드캐스팅 규칙을 따릅니다.

- 입력 배열은 동일한 차원 수를 가질 필요가 없습니다.
    - 결과 배열은 가장 큰 차원을 가진 입력 배열과 동일한 차원 수를 가집니다.
    - 각 차원의 크기는 해당 차원의 가장 큰 크기입니다.
    - 누락된 차원은 크기가 1인 것으로 간주됩니다.
- 형태 비교는 가장 오른쪽 차원에서 시작하여 왼쪽으로 진행됩니다.
- 두 차원은 크기가 같거나 그 중 하나가 1인 경우 호환됩니다.

**Qiskit 에서:**
- 파라미터 값은 n × m 배열로 제공됩니다
- 관측가능량은 k × 1 배열로 제공됩니다

그런 다음 둘 다 표준 브로드캐스팅 규칙에 따라 결합됩니다

```python
simulator = AerSimulator()

# Parameterized 1-qubit circuit
theta = Parameter("θ")
qc = QuantumCircuit(1)
qc.rx(theta, 0)

# Estimator primitive
estimator = Estimator(mode=simulator)

pm = generate_preset_pass_manager(optimization_level=1, backend=simulator)
transpiled_circuit = pm.run(qc)
transpiled_circuit.draw('mpl')
```

**Single observable and parameter set**

```python
# Broadcasting Example 1: Single observable and parameter set
#Parameter values:   (5 × 1)
#Observables:        (1 × 1)
#--------------------------------
#Output:             (5 × 1)

parameter_values = [[v] for v in np.linspace(0, np.pi, 5)]
observable = SparsePauliOp("Z")
estimator_pub = (transpiled_circuit, observable, parameter_values)
job = estimator.run([estimator_pub])
result = job.result()
print(result[0].data.evs)
print(f"Output shape: {result[0].data.evs.shape}")
```

### 4. 작업 세션 실행

세션은 동일한 양자 하드웨어에서 여러 관련 작업을 실행하는 효율적인 방법을 제공합니다. 세션은 작업 간에 컨텍스트를 유지하여 오버헤드를 줄이고 반복적 알고리즘의 성능을 향상시킬 수 있습니다.

**세션**은 다음과 같은 컨텍스트입니다:
1. 여러 작업을 위한 하드웨어 시간 예약
2. 작업 간 회로 컨텍스트 유지
3. 반복적 알고리즘을 위한 오버헤드 감소
4. 관련 작업을 위한 일관된 하드웨어 조건 보장

**세션 라이프사이클:**
1. **열기 (Open)**: 특정 백엔드로 세션 생성
2. **사용 (Use)**: 세션 내에서 여러 작업 실행
3. **닫기 (Close)**: 명시적으로 닫거나 타임아웃 허용

**주요 파라미터:**
- **TTL (Time To Live)**: 최대 세션 지속 시간 (기본값: 8 시간)
- **Max Time**: 세션 내 최대 작업 실행 시간



### 6. 세션 길이

`Batch` 와 유사하게, 세션의 최대 TTL 은 `max_time` 으로 정의할 수 있으며, 이는 세션이 시작될 때부터 시작됩니다. 최대 시간에 도달하면 세션이 닫히고 실행 중인 작업은 완료되지만 대기 중인 모든 작업은 실패합니다.

| 플랜 유형 | 최대 작업 지속 시간 |
|-----------|-------------------|
| 프리미엄 플랜 | 8 시간 |

또한 구성할 수 없는 인터랙티브 TTL(1 분) 도 있으며, 이 시간 내에 세션 작업이 큐에 없으면 세션은 일시적으로 비활성화됩니다.

### 7. 세션 종료

다음 중 하나가 발생하면 세션이 닫힙니다:

* TTL 에 도달한 경우
* 세션이 수동으로 취소되거나 `session.close()` 를 사용하여 닫힌 경우
* 세션이 컨텍스트 매니저에서 사용되어 컨텍스트가 끝날 때 자동으로 닫힌 경우

세션이 닫히면 더 이상 작업을 수락하지 않지만, 이미 제출된 작업은 실행됩니다.

```python
session = Session(backend=backend)
 
estimator = Estimator(mode=session)
sampler = Sampler(mode=session)
job1 = estimator.run([estimator_pub])
job2 = sampler.run([sampler_pub])
print(f"Result1: {job1.result()}")
print(f"Result2: {job2.result()}")
 
# Manually close the session. Running and queued jobs will run to completion.
session.close()
```

### 8. 세션 상태 확인

`session.status()` 를 사용하여 세션 상태를 확인할 수 있으며, 상태는 다음 중 하나일 수 있습니다:

* Pending (대기 중)
* In Progress, accepting new jobs (진행 중, 새 작업 수락 중)
* In Progress, not accepting new jobs: 제출된 작업이 완료되고 그 후 세션이 닫힘 (진행 중, 새 작업 수락 안 함)
* Closed (닫힘)

### 11. 요약

이 노트북에서는 다음을 다뤘습니다:

실제 하드웨어에서 회로 실행 시연:

1. **실제 하드웨어에서의 실행** 은 인증, 백엔드 선택, 회로 준비, 작업 제출, 결과 조회를 포함합니다.
2. **Qiskit 런타임 프리미티브** 는 하드웨어 특화적인 세부사항을 추상화하는 표준 인터페이스입니다.
3. **샘플러** 는 회로의 측정값 또는 카운트를 반환하고, **에스티메이터** 는 관측가능량의 기댓값을 반환합니다.
4. **브로드캐스팅 규칙** 은 다양한 관측가능량 그룹과 파라미터 값에 대한 기댓값을 계산하는 데 사용됩니다.
5. **세션 모드** 는 독점적인 QPU 접근을 제공하며, 고전적 및 양자 정보가 필요한 반복적 워크플로우에 가장 적합하게 사용됩니다.
