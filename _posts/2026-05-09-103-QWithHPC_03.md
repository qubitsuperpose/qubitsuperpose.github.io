# 3. Programming models

프로그래밍 모델은 소프트웨어가 어떻게 구조화되고 실행되는지를 정의하는 기본 사양입니다. 개발자들이 알고리즘을 표현하고 코드를 조직할 수 있는 프레임워크를 제공하며, 종종 하드웨어 또는 실행 환경의 저수준 세부 사항을 추상화합니다. 서로 다른 모델은 서로 다른 유형의 문제와 하드웨어 아키텍처에 적합하며, 다양한 수준의 추상화와 제어를 제공합니다.

이번 강의에서는 양자 및 고전 프로그래밍 모델을 검토하고, 이들을 결합하여 이종 환경에서 알고리즘을 작동시키는 방법을 살펴보겠습니다. Iskandar Sitdikov가 다음 영상에서 개요를 제공합니다.

## 1. QPUs를 위한 프로그래밍 모델
양자 컴퓨터를 위한 프로그래밍 모델부터 시작하겠습니다. 거의 모든 양자 개발자에게 친숙한 기본 프로그래밍 모델은 양자 회로(quantum circuit)입니다. 여기서는 양자 회로 모델의 세부 사항을 다루지 않겠습니다. 이미 John Watrous의 훌륭한 [강의](/learning/courses/basics-of-quantum-information/quantum-circuits/introduction)가 이를 자세히 설명하고 있기 때문입니다. 다만, 회로는 큐비트를 나타내는 선(와이어)들, 양자 상태에 대한 연산을 나타내는 게이트들, 그리고 측정들로 구성된다는 점만 언급하겠습니다.

![A quantum circuit diagram showing qubits as horizontal lines and quantum gates as boxes or connections between qubits.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/circuit.svg)

양자 컴퓨팅에서 또 다른 중요한 프로그래밍 모델 개념은 **computational primitives**입니다. 이 프리미티브들은 사용자가 양자 컴퓨터로 수행하고자 하는 가장 일반적인 작업들을 나타냅니다. 현재 여러 프리미티브가 제공되며, 그중 [Executor](/docs/guides/directed-execution-model#executor-primitive)도 포함됩니다. 이 강의에서는 주로 Sampler와 Estimator 프리미티브에 초점을 맞춥니다. Sampler는 양자 회로로 준비된 상태를 샘플링할 수 있게 해줍니다. 즉, 양자 회로에서 준비된 양자 상태를 구성하는 계산 기저 상태들이 무엇인지를 알려줍니다. Estimator는 양자 회로로 준비된 상태에 있는 시스템의 관측량(expectation value)을 추정할 수 있게 해줍니다. 흔한 예로는 특정 상태에 있는 시스템의 에너지를 추정하는 것입니다.

![A model histogram of results from sampler. Some states are very likely to be measured, others are very unlikely.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/sampler.svg)

이 섹션에서 마지막으로 다룰 내용은 **transpilation**입니다. Transpilation은 주어진 입력 회로를 특정 양자 장치의 물리적 제약과 명령어 집합 아키텍처(ISA)에 맞게 재작성하는 과정입니다. 고전 컴파일러와 유사하게, 이는 추상적인 유니터리 연산을 대상 장치가 실행할 수 있는 네이티브 게이트 집합으로 변환하는 것을 의미합니다. 또한 잡음이 있는 양자 컴퓨터에서 효율적으로 실행되도록 회로 명령어를 최적화하며, 여러 최적화 단계를 적용하면서 점차 회로 구조를 변경합니다.

![A diagram of transpilation showing how an abstract circuit is mapped into an instruction set architecture circuit. That is, the circuit is rewritten using the native gates and connectivity of the target hardware.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/transpilation.svg)

## 2. Classical programming models
**고전 프로그래밍 모델**

고전 컴퓨터를 위한 프로그래밍 모델은 많지만, 이 섹션에서는 가장 인기 있는 두 가지 모델에 초점을 맞춥니다: 병렬 프로그래밍과 태스크 워크플로우. 이러한 두 모델을 양자 프로그래밍 모델과 함께 사용하면 거의 모든 복잡한 하이브리드 양자-고전 워크플로우를 표현할 수 있습니다.

### 2.1 병렬 프로그래밍

병렬 프로그래밍은 프로그램을 동시에 실행될 수 있는 하위 문제로 나누는 모델입니다. 병렬 프로그래밍에는 두 가지 주요 패러다임이 있습니다:

* 공유 메모리 병렬 처리 (Open Multiprocessing, 또는 OpenMP): 단일 컴퓨트 노드 내의 여러 코어를 활용하는 데 사용됩니다. 실행 스레드들이 단일 메모리 공간을 공유합니다.

* 분산 메모리 병렬 처리 (Message Passing Interface, 또는 MPI): 여러 개별 컴퓨트 노드에 걸쳐 확장하는 데 사용됩니다. 각 프로세스는 자체 격리된 메모리 공간을 가집니다.

여기서는 다중 노드 슈퍼컴퓨팅과 대규모 이기종 양자-고전 작업을 조정하는 데 필수적인 분산 메모리 모델에 초점을 맞춥니다.

분산 메모리 병렬 프로그래밍 모델에서 작동하기 위해 이해해야 할 몇 가지 개념이 있습니다:

* 프로세스 - 자체 메모리 공간을 가진 프로그램의 독립적인 인스턴스.
* 랭크 - 각 프로세스에 할당되는 고유한 정수 식별자. 통신 중 송신자와 수신자를 식별하는 데 특별히 사용됩니다(우선순위를 의미하는 "랭크"는 아닙니다).
* 동기화 - 서로 다른 랭크와 프로세스 간의 조정을 위한 메커니즘.
* 단일 프로그램, 다중 데이터 (SPMD) - 단일 소스 코드 인스턴스가 여러 프로세스에서 동시에 실행되며, 각각 전체 데이터의 서로 다른 하위 집합에서 작동하는 추상적인 계산 모델.
* 메시지 전달 - 분산 메모리 아키텍처에서 사용되는 통신 패러다임으로, 독립적인 프로세스가 데이터와 중간 결과를 교환할 수 있게 합니다. 서로 다른 컴퓨트 노드 간 실행을 조정하기 위해 명시적인 'send'와 'receive' 연산에 의존합니다.

이 메시지 전달 패러다임을 병렬 아키텍처에 구현하는 MPI라는 표준이 있습니다. MPI는 위에 나열된 모든 개념의 기능적 구현체로, 프로세스를 관리하고, 랭크를 할당하고, 동기화를 촉진하며, SPMD 모델 하에서 메시지 전달을 가능하게 하는 구체적인 라이브러리 호출을 제공합니다. 이러한 모든 개념을 종합하면, 병렬 프로그램의 실행은 다음과 같은 방식으로 이루어진다고 할 수 있습니다:

* 단일 컴파일된 프로그램(동일한 바이너리 파일)이 작업 런처에 의해 복사되어 여러 노드에 걸쳐 여러 병렬 프로세스를 생성하여 실행됩니다.
* 프로그램의 주요 제어 흐름은 프로세스의 랭크에 의해 결정됩니다. 이것이 SPMD 원리가 작동하는 방식입니다: 프로그램은 조건부 논리(예: if (rank == 0))를 사용하여 워커 프로세스가 특정 병렬화된 코드 섹션만 실행하도록 하고, 마스터 프로세스(종종 Rank 0)가 초기화와 최종 집계를 처리합니다.
* 프로세스 간 통신은 메시지 전달(MPI 사용)을 통해 발생하며, 프로세스가 다른 랭크와 데이터나 중간 결과를 교환해야 할 때마다 호출됩니다.

시각적으로는 다음과 같이 보일 것입니다:

![A diagram of a task being divided between nodes.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/distributed-program.svg)

방금 배운 개념 중 일부를 코드에 적용해 보겠습니다.

먼저, OpenMPI를 사용하여 간단한 "hello world" 병렬 프로그램을 실행해 보겠습니다. OpenMPI는 병렬 프로그래밍에서 메시지 전달을 위한 표준인 MPI 프로토콜의 구현입니다. 여기서는 MPI 표준에 대한 Python 바인딩인 mpi4py Python 패키지를 사용합니다.

```shell
$ vim mpi-hello-world.py
```

```python
from mpi4py import MPI
import sys

comm = MPI.COMM_WORLD
rank = comm.Get_rank()
size = comm.Get_size()

sys.stdout.write(f"[Rank {rank}] Hello from process {rank} of {size}!\n")

if rank == 0:
    data = {'answer': 42, 'pi': 3.14}
    sys.stdout.write(f"[Rank {rank}] Sending: {data}\n")
    comm.send(data, dest=1, tag=42)
elif rank == 1:
    data = comm.recv(source=0, tag=42)
    sys.stdout.write(f"[Rank {rank}] Received: {data}\n")

~
~
```

본 프로그램을 실행하기 위해 두 개의 노드를 사용할 예정이며, 어떤 노드를 사용할지는 제출 스크립트에 명시하겠습니다.

```shell
$ vim mpi-hello-world.sh

#!/bin/bash
#
#SBATCH --job-name=mpi-hello-world
#SBATCH --output=mpi-hello-world.out
#SBATCH --nodes=2
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1
#SBATCH --partition=normal

/usr/lib64/openmpi/bin/mpirun python /data/ch3/parallel/mpi-hello-world.py
```

그다음 셸 스크립트를 실행하세요.

```shell
$ sbatch mpi-hello-world.sh
```

작업 결과 로그를 확인할 수 있습니다.

```shell
$ cat mpi-hello-world.out | grep Rank

[Rank 1] Hello from process 1 of 2!
[Rank 0] Hello from process 0 of 2!
[Rank 0] Sending: {'answer': 42, 'pi': 3.14}
[Rank 1] Received: {'answer': 42, 'pi': 3.14}
```

여기서는 두 개의 노드를 사용했으며, 각 노드의 프로세스는 랭크 0과 랭크 1이라는 순위로 식별됩니다. 이 순위는 프로그램 제어 흐름을 결정하는 데 사용됩니다.


## 3. Task workflows

이제 Task workflow 프로그래밍 모델에 대해 이야기해 보겠습니다. Task workflow는 계산을 방향성 비순환 그래프(DAG)로 추상화합니다. 이 그래프에서 각 노드는 특정 task나 job을 나타내고, 에지(노드들을 연결하는 화살표)는 그들 사이의 의존성(데이터와 순서)을 나타냅니다. Scheduler는 task를 자원에 매핑하고 실행을 조율하는 컴포넌트입니다.

양자 컴퓨팅에 적용된 task workflow 모델의 구체적인 예는 [Qiskit patterns framework](https://quantum.cloud.ibm.com/docs/guides/intro-to-patterns)입니다. Qiskit pattern은 도메인별 문제를 특히 양자 task를 위한 일련의 단계로 분해하도록 설계된 일반적인 프레임워크입니다. 이는 IBM Quantum® 연구자(및 타인)들이 개발한 새로운 기능들의 원활한 조합성을 허용하며, 양자 컴퓨팅 task가 강력한 이기종(CPU/GPU/QPU) 컴퓨팅 인프라에 의해 수행되는 미래를 가능하게 합니다. Qiskit pattern의 네 단계는 mapping, optimization, execution, post-processing이며, 모든 task가 파이프라인에서 순차적으로 하나씩 실행됩니다. 그러나 task workflow에서는 선형 실행 순서에 제한되지 않고 task를 병렬로 실행할 수 있습니다. Workflow의 각 task는 그 자체로 전체 병렬 job이 될 수 있습니다. 따라서 이러한 모델들을 자유롭게 조합하여 임의로 복잡한 알고리즘을 기술할 수 있으며, Slurm과 같은 workload manager가 이를 처리합니다.

![A diagram of computing tasks organized into a workflow in which some processes are executed in parallel and others in sequence.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/workflow.svg)

위 이미지는 Qiskit pattern이 작동하는 모습을 보여줍니다. Workflow는 네 단계로 구성된 그래프 구조를 가지고 있습니다. 이 가지 형태의 구조는 scheduler에 의해 조율되고 실행됩니다. 문제는 초기 단계에서 양자 실행 가능한 형태(양자 회로)로 매핑됩니다. 다음 단계에서는 이 양자 회로가 특정 양자 하드웨어에 맞게 최적화됩니다. 이미지는 이를 병렬 프로세스로 보여주며, 동시에 여러 최적화 전략이 적용될 수 있음을 나타냅니다. 최적화된 양자 회로는 실제 양자 하드웨어에서 실행됩니다. 이는 이미지의 세 번째 단계로, scheduler가 하나의 보라색 양자 처리 장치와 함께 작동하는 모습입니다. 마지막으로, 결과는 고전적 자원에 의해 post-processing됩니다.

### 3.1 왜 둘 다 필요한가?

그렇다면 왜 병렬 프로그래밍과 태스크 워크플로우를 모두 필요로 하는가? 양자 병렬성에 대한 많은 논의에도 불구하고, 양자 컴퓨팅에서 모든 것이 병렬인 것은 아니라는 점을 명확히 할 필요가 있다.

SQD 워크플로우에 대한 이전 수업에서는 병렬화할 수 없는 몇 가지 프로세스를 언급했다. 예를 들어, 우리는 매트릭스를 다루기 쉬운 차원의 부분 공간으로 투영하기 위해 많은 양자 측정 결과를 필요로 한다. 차례로, 우리는 양자 측정의 자기 일관성을 확인하기 위해 대각화된 매트릭스와 관련된 상태 벡터가 필요하다(예를 들어, 전하 보존을 사용). 이 모든 후에, 우리는 기저 상태 에너지가 우리의 목적에 충분히 수렴했는지 결정해야 한다. 이러한 단계들은 필연적으로 순차적이며, 진행하기 전에 수렴과 자기 일관성 조건을 테스트해야 한다.

![A schematic of the workflow specific to sample-based quantum diagonalization. The steps include a variational quantum circuit, using measurements to project the Hamiltonian into a subspace, then using a classical optimizer to update variational parameters in the circuit and repeating.](https://quantum.cloud.ibm.com/learning/images/courses/integrating-quantum-and-high-performance-computing/programming-models/sqd-workflow.svg)

이 워크플로우는 다음 섹션에서 더 자세히 다루고 구현될 것이다. 이 섹션에서 알아가야 할 유일한 점은 태스크 워크플로우가 필요하다는 것이다.

## 4. 프로그래밍 실습

프로그래밍 모델의 장점은 모든 모델을 자유롭게 조합하여 사용할 수 있다는 것입니다. 양자 및 고전 프로그래밍 모델을 알면 임의의 복잡성을 가진 이기종 연산을 설계하고 하드웨어에서 실행할 수 있습니다. 지난 장에서 배운 Qiskit 패턴(맵핑, 최적화, 실행, 후처리)을 Slurm 내에서 구현한 간단한 워크플로우 예제를 통해 이를 연습해 보겠습니다. 네 가지 작업은 각각 별도의 Slurm 작업으로 실행되며, 각 작업마다 고유한 리소스를 갖습니다. 최적화 작업은 MPI를 사용하여 회로를 병렬로 최적화합니다(위 이미지와 같이 예시를 위한 것입니다). 실행 작업은 양자 리소스와 양자 프로그래밍 모델(회로 및 샘플러)을 사용합니다. 마지막 작업인 후처리는 다시 MPI를 사용하여 고전 리소스를 병렬로 활용합니다.

### 4.1 매핑

`mapping.py` 프로그램은 양자 머신러닝 및 양자 벤치마크 문헌에서 자주 사용되는 `PauliTwoDesign` 회로를 구축하도록 설계되었습니다. 이 회로는 무작위 초기 매개변수를 가진 n-1번째 큐비트 시스템의 Z 방향에서 (n-1)번째 큐비트를 측정하는 간단한 관측량을 사용합니다. 양자 회로(qasm 파일로 변환됨), 관측량, 그리고 매개변수는 각각 `data` 디렉터리 아래에 별도의 파일로 저장되며, 최적화 단계의 입력으로 사용됩니다.

이 단계의 셸 스크립트(`mapping.sh`)는 다음과 같습니다.

```shell
#!/bin/bash
#
#SBATCH --job-name=mapping
#SBATCH --output=mapping.out
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1
#SBATCH --partition=normal


srun python /data/ch3/workflows/mapping.py
```

이는 작업 이름, 출력 형식 및 노드/태스크/CPU 수를 정의합니다.


### 4.2 최적화

`optimization.py` 프로그램은 매핑 단계에서 파일을 가져오는 것으로 시작합니다. 여기서는 `QRMI`를 사용하여 양자 리소스를 프로그램으로 가져옵니다.

```python
qrmi = QRMI()
resources = qrmi.resources()
quantum_resource = resources[0]
...
```

그다음 `optimization_level=1`로 설정하여 양자 회로를 트랜스파일링하고 회로 레이아웃을 관측 가능한 데이터에 적용하는 간단한 최적화를 수행한 후, 이 결과를 데이터 폴더에 저장합니다.

이 단계의 셸 스크립트(`optimization.sh`)는 다음과 같습니다.

```shell
#!/bin/bash
#SBATCH --job-name=optimization
#SBATCH --output=output/optimization.out
#SBATCH --ntasks=4
#SBATCH --partition=classical

srun python3 /tmp/optimization.py
```

여기서 `--ntasks=4`는 Slurm에서 병렬 프로세스를 위해 4개의 클래식 태스크를 요청합니다.


### 4.3 실행

이 단계는 이전 단계에서 최적화된 양자 회로를 Estimator를 사용하여 QPU에서 실행하는 핵심 양자 단계입니다. 이를 위해 먼저 트랜스파일된 양자 회로, 관측량, 초기 매개변수 세 파일을 Estimator에 전달합니다. Estimator는 관측량의 예상값을 계산하여 출력합니다.

`execution.sh` 스크립트는 Slurm 플러그인을 활용하여 양자 리소스를 사용합니다.

```
#!/bin/bash
#
#SBATCH --job-name=execution
#SBATCH --output=execution.out
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=1
#SBATCH --cpus-per-task=1
#SBATCH --partition=quantum
#SBATCH --gres=qpu:1


srun python /data/ch3/workflows/execution.py
```



### 4.4 후처리

후처리 단계에는 일반적으로 고전적인 대각화 및 자체 일관성 검사가 포함됩니다. 또한 반복적인 과정을 거칠 수도 있습니다. 물리적 맥락과 반복 단계의 목적이 명확해지는 다음 강의에서 후처리 단계를 자세히 살펴보는 것이 가장 유용합니다.



### 4.5 모든 작업을 하나로 통합하기

`sbatch` 명령어에 종속성 인수를 사용하면 이러한 모든 작업을 워크플로로 연결할 수 있습니다.

```shell
$ MAPPING_JOB=$(sbatch --parsable mapping.sh)
$ OPTIMIZE_JOB=$(sbatch --parsable --dependency=afterok:$MAPPING_JOB optimization.sh)
$ EXECUTE_JOB=$(sbatch --parsable --dependency=afterok:$OPTIMIZE_JOB execute.sh)

```

그리고 우리는 Slurm 실행 큐를 확인할 수 있습니다.

```shell
$ squeue
#             JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
#                 3 classical  mapping    admin PD       0:00      1 (None)
#                 4 classical optimiza    admin PD       0:00      1 (Dependency)
#                 5   quantum  execute    admin PD       0:00      1 (Dependency)
```

이것은 다양한 프로그래밍 모델을 보여주기 위한 간단한 예시였습니다. 다음 장에서는 실제 알고리즘을 살펴보고 유용한 워크플로우에서 프로그래밍 모델과 리소스 관리를 시연해 보겠습니다.



## 5. 요약
